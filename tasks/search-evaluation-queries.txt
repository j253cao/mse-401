Search evaluation query bank and test plan
==========================================

How this maps to the app
------------------------

- The main text box on RecommendationPage calls api.recommend([search], filters), which hits POST /recommend and ranks courses via embeddings + filters (backend recommender).
- For fair comparison across queries, keep filters fixed for a session (e.g. same program, incoming level, department toggles, exploration on/off) and only change the query string. Repeat the same query with filters toggled if you care about dependency / exploration behavior.

Flow: Natural language query -> RecommendationPage -> POST /recommend -> get_recommendations

Stress / edge queries
---------------------

Very short:
  math, design, energy, AI

Multi-intent:
  I want to learn C++ and also read about philosophy

Jargon mismatch:
  how do I build a website (may pull CS or broad software courses)

Typo / informal:
  thermo dyamics, lin alg

Non-course intent:
  co-op interview tips (expect weak or noisy matches—good for calibration)


Questions to ask for each query (quick rubric)
----------------------------------------------

Score subjectively (e.g. 1–5) or pass/fail.

1. Top-3 relevance: Are the top three results clearly related to what you meant?
2. False positives: Any highly ranked course that feels clearly off? Note the course code.
3. Coverage: Did something you know exists at UW for this topic fail to appear in the top ~15?
4. Title vs description: Does ranking seem driven by buzzwords in the title vs actual syllabus fit?
5. Filters interaction: If you turn exploration / department filters on or off, does ranking stay sensible?

Optional columns for a spreadsheet:
  query_id, query_text, filter_snapshot, top1_code, top5_notes, score_relevance, notes
